# MVP Checklist

## Назначение

Этот документ нужен для финальной ручной проверки MVP перед тем, как считать его закрытым.

Проверка должна подтвердить `Definition of Done` из `docs/IMPLEMENTATION_PLAN.md` и `docs/MVP.md` на реальном сценарии с двумя Obsidian vault.

---

## Предусловия

- сервер запускается локально
- plugin собран
- есть два отдельных локальных vault:
  - `vault-a`
  - `vault-b`
- plugin установлен и включён в обоих vault
- в обоих vault одинаковый `Server URL`
- в обоих vault одинаковый `Vault ID`
- в обоих vault разные `Device ID`
- в обоих vault одинаковый `Auth token`
- `Poll interval` выставлен в `2`
- `Auto sync` включён

---

## Запуск

### Сервер

```bash
./run-server.sh
```

### Plugin

```bash
cd plugin
npm install
npm run build
```

Собранные `main.js` и `manifest.json` должны быть установлены в оба test vault.

## Чеклист

### 1. Новый файл

Шаги:

1. В `vault-a` создать `notes/mvp-new.md`
2. Подождать polling или вызвать `Sync now`
3. Открыть `vault-b`

Ожидание:

- файл появляется в `vault-b`
- содержимое совпадает

---

### 2. Изменение существующего файла

Шаги:

1. В `vault-a` изменить `notes/mvp-new.md`
2. Подождать polling или вызвать `Sync now`
3. Открыть `vault-b`

Ожидание:

- изменённое содержимое доезжает в `vault-b`
- вторая копия файла не создаётся

---

### 3. Удаление файла

Шаги:

1. В `vault-a` удалить `notes/mvp-new.md`
2. Подождать polling или вызвать `Sync now`
3. Открыть `vault-b`

Ожидание:

- файл исчезает из `vault-b`
- повторный sync не возвращает удалённый файл

---

### 4. Конфликт

Шаги:

1. Создать общий файл `notes/mvp-conflict.md`
2. Дождаться, пока он появится в обоих vault
3. Отключить сеть или остановить сервер
4. Изменить файл по-разному в `vault-a` и `vault-b`
5. Запустить сервер обратно
6. Сначала синхронизировать `vault-a`
7. Затем синхронизировать `vault-b`

Ожидание:

- в `vault-b` появляется `notes/mvp-conflict (conflict).md`
- основной `notes/mvp-conflict.md` содержит серверную версию
- локальная версия из `vault-b` не потеряна

---

### 5. Рестарт клиента

Шаги:

1. Выполнить успешную синхронизацию в обоих vault
2. Полностью закрыть Obsidian
3. Открыть его снова
4. Запустить `Sync now`

Ожидание:

- plugin стартует без сброса состояния
- уже синхронизированные файлы не переливаются заново без причины
- новые изменения после рестарта продолжают синхронизироваться

---

### 6. Догоняющая синхронизация после офлайна

Шаги:

1. Остановить сервер
2. В `vault-a` создать или изменить несколько файлов
3. Убедиться, что sync временно падает
4. Запустить сервер обратно
5. Дождаться следующего polling цикла или вызвать `Sync now`
6. Открыть `vault-b`

Ожидание:

- изменения из `vault-a` доходят после восстановления сети
- локальные данные не теряются
- plugin продолжает работать без ручного сброса state

---

## Критерий завершения

MVP можно считать закрытым, когда все шесть сценариев выше проходят без ручного вмешательства в state, базу сервера или файлы plugin data.

---

## Что делать при сбое

Если сценарий не проходит, зафиксировать:

- какой шаг воспроизводит проблему
- какой vault был источником изменения
- что ожидалось
- что получилось фактически
- были ли conflict copies
- были ли ошибки в console или Notice
