# MVP Checklist

## Назначение

Этот документ теперь используется как ручной регрессионный чеклист для текущей версии проекта,
а не только для исторического MVP.

Он покрывает:

- базовый sync
- vault registry
- device registry
- E2EE
- history/restore

---

## Предусловия

- сервер запускается локально
- plugin собран
- есть два локальных Obsidian vault:
  - `vault-a`
  - `vault-b`
- plugin установлен и включён в обоих vault
- в обоих vault одинаковые:
  - `Server URL`
  - `Auth token`
  - `vaultId`
  - E2EE passphrase
- в обоих vault разные `Device ID`
- `Poll interval = 2`
- `Auto sync` включён

---

## Запуск

### Сервер

```bash
cp server/.env.example server/.env
./run-server.sh
```

### Plugin

```bash
cd plugin
npm install
npm run build
```

Собранные `main.js`, `manifest.json` и при необходимости `styles.css` должны быть установлены
в оба test vault.

## Чеклист

### 1. Создание vault и подключение папки

Шаги:

1. В `vault-a` открыть настройки plugin.
2. Ввести `Server URL` и `Auth token`.
3. Нажать `Check`.
4. Создать новый vault через `Create vault`.

Ожидание:

- vault появляется в серверном реестре
- текущая папка подключается к этому `vaultId`
- plugin не показывает ошибку авторизации

### 2. Новый файл

Шаги:

1. В `vault-a` создать `notes/mvp-new.md`
2. Подождать realtime/polling или вызвать `Sync now`
3. Открыть `vault-b` и присоединить его к тому же vault

Ожидание:

- файл появляется в `vault-b`
- содержимое совпадает

### 3. Изменение существующего файла

Шаги:

1. В `vault-a` изменить `notes/mvp-new.md`
2. Дождаться sync
3. Проверить `vault-b`

Ожидание:

- изменённое содержимое доезжает в `vault-b`
- вторая копия файла не создаётся

### 4. Удаление файла

Шаги:

1. В `vault-a` удалить `notes/mvp-new.md`
2. Дождаться sync
3. Проверить `vault-b`

Ожидание:

- файл исчезает из `vault-b`
- повторный sync не возвращает его обратно

### 5. Конфликт

Шаги:

1. Создать общий файл `notes/mvp-conflict.md`
2. Дождаться, пока он появится в обоих vault
3. Остановить сервер или временно разорвать сеть
4. Изменить файл по-разному в `vault-a` и `vault-b`
5. Вернуть сервер
6. Сначала синхронизировать `vault-a`
7. Затем синхронизировать `vault-b`

Ожидание:

- в `vault-b` появляется conflict copy
- основной файл содержит серверную версию
- локальная версия из `vault-b` не теряется

### 6. Device registry

Шаги:

1. После успешного sync на двух клиентах открыть `Refresh devices` в одном из них.

Ожидание:

- сервер видит оба `Device ID`
- у устройств есть `first_seen_at` и `last_seen_at`

### 7. E2EE

Шаги:

1. Создать зашифрованный vault через `Create vault`.
2. Создать в `vault-a` новый markdown-файл.
3. Дождаться sync.
4. Подключить `vault-b` к тому же vault с той же passphrase.

Ожидание:

- файл успешно расшифровывается на `vault-b`
- при неверной passphrase plugin не должен молча показывать испорченный текст
- plugin должен сигнализировать о проблеме с decrypt/passphrase

### 8. History и restore

Шаги:

1. В `vault-a` несколько раз изменить один и тот же файл.
2. Открыть его и вызвать `Show active file server history`.
3. Выбрать старую версию и выполнить restore.
4. Дождаться sync на `vault-b`.

Ожидание:

- история показывает несколько версий
- restore создаёт новую актуальную версию
- восстановленное содержимое доезжает до `vault-b`

### 9. Рестарт клиента

Шаги:

1. Выполнить успешную синхронизацию.
2. Полностью закрыть Obsidian.
3. Открыть его снова.
4. Запустить `Sync now`.

Ожидание:

- plugin стартует без потери sync state
- уже синхронизированные файлы не переливаются заново без причины
- синхронизация продолжает работать

### 10. Догоняющая синхронизация после офлайна

Шаги:

1. Остановить сервер.
2. В `vault-a` создать или изменить несколько файлов.
3. Убедиться, что sync временно падает.
4. Запустить сервер обратно.
5. Дождаться следующего цикла или вызвать `Sync now`.
6. Проверить `vault-b`.

Ожидание:

- изменения из `vault-a` доходят после восстановления сети
- локальные данные не теряются
- plugin продолжает работать без ручного редактирования state

## Что фиксировать при сбое

- какой шаг воспроизводит проблему
- какой vault был источником изменения
- какой `Device ID` участвовал
- что ожидалось
- что получилось фактически
- были ли conflict copies
- были ли ошибки в Notice, devtools или серверных логах
