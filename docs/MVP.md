# MVP

## Назначение

Этот документ фиксирует первую рабочую версию sync-системы. Его задача:

- ограничить scope
- убрать двусмысленность перед началом реализации
- дать чёткий критерий, что считается завершённым MVP

---

## Цель MVP

Сделать рабочую синхронизацию одного Obsidian vault между несколькими устройствами с сохранением конфликтов и устойчивым локальным состоянием.

MVP должен быть пригоден для реального тестирования на небольшом наборе Markdown-файлов и вложений.

---

## Что входит в MVP

- один сервер синхронизации
- один пользовательский сценарий без сложной multi-tenant модели
- один vault на пользователя
- синхронизация между несколькими клиентами
- whole-file sync
- polling-based sync loop
- локальная SQLite БД состояния на клиенте
- серверное версионирование файлов
- conflict detection через `base_version`
- conflict copy при расхождении версий
- tombstones для удалений
- Rust-сервер на `axum`

---

## Что не входит в MVP

- E2EE
- history/restore UI
- merge содержимого файлов
- realtime push через WebSocket/SSE
- selective sync
- chunked sync
- deduplication blobs
- rate limiting
- полноценный auth flow
- device approval/revoke
- rename detection как отдельная модель

---

## Продуктовые допущения

### Vault model

- один клиент работает с одним локальным vault
- сервер хранит метаданные и актуальное состояние файла
- сервер является `source of truth` по версиям

### File model

- файл идентифицируется по относительному пути внутри vault
- путь должен быть нормализован и безопасен
- `rename` в MVP трактуется как `delete + create`

### Conflict model

Если локальный клиент пытается записать файл по устаревшей версии:

- сервер отклоняет запись
- клиент получает конфликт
- локальная версия сохраняется как `filename (conflict).ext`
- серверная версия применяется как основная

### Delete model

Удаления не считаются простым исчезновением файла.

Удаление должно:

- попадать на сервер как отдельное событие
- повышать версию файла
- отражаться в change feed
- доходить до остальных клиентов через tombstone

## Пользовательские сценарии, которые обязаны работать

### Сценарий 1. Новый файл

1. Пользователь создаёт новый файл в vault на устройстве A
2. Клиент A обнаруживает новый файл
3. Файл загружается на сервер
4. Устройство B получает изменение через change feed
5. Файл появляется в vault на устройстве B

### Сценарий 2. Изменение файла

1. На устройстве A меняется содержимое существующего файла
2. Клиент A загружает новую версию
3. Сервер повышает версию файла
4. Устройство B скачивает новую версию

### Сценарий 3. Удаление файла

1. Пользователь удаляет файл на устройстве A
2. Клиент A отправляет delete на сервер
3. Сервер создаёт tombstone
4. Устройство B получает tombstone через change feed
5. Файл удаляется локально на устройстве B

### Сценарий 4. Конфликт

1. Устройства A и B изменяют один и тот же файл параллельно
2. Устройство A успевает загрузить версию раньше
3. Устройство B загружает файл с устаревшим `base_version`
4. Сервер возвращает conflict
5. На устройстве B локальная версия сохраняется как conflict copy
6. Серверная версия становится основной

### Сценарий 5. Оффлайн догоняющая синхронизация

1. Устройство работает без сети
2. Локальные изменения сохраняются в vault
3. После восстановления сети клиент догоняет сервер
4. Изменения синхронизируются без потери локального состояния

## Технические требования MVP

### Обязательно

- хэширование файлов по содержимому
- SQLite state на клиенте
- SQLite metadata на сервере
- безопасная нормализация путей
- защита от path traversal
- atomic file writes
- устойчивость к рестарту клиента
- global change feed на сервере

### Допустимые упрощения

- poll раз в несколько секунд вместо push
- binary attachments синкаются как обычные файлы
- без оптимизации трафика для больших файлов
- single-node сервер без горизонтального масштабирования

---

## Нефункциональные границы

- приоритет у корректности данных, а не у максимальной скорости
- приоритет у явных конфликтов, а не у автоматического merge
- приоритет у простой архитектуры, а не у полной feature parity с Obsidian Sync

---

## Что считаем успехом MVP

MVP считается готовым, если:

- два клиента могут синхронизировать один vault через сервер
- новые файлы доезжают между устройствами
- изменения файлов доезжают между устройствами
- удаления доезжают между устройствами
- конфликт не приводит к потере локальной версии
- состояние синка не теряется после перезапуска клиента
- клиент может догнать сервер после временного отсутствия сети
